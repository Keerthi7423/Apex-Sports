SECTION 4: RABBITMQ — REPLACING REDIS PUB/SUB
================================================================

RabbitMQ is a production-grade message broker used by
companies like Instagram, Reddit, and Zalando.
It is more powerful than Redis pub/sub and shows deeper
knowledge of distributed systems.

Why RabbitMQ over Redis pub/sub:
  Persistent messages — if notification service is down,
    messages are queued and delivered when it comes back up
  Message acknowledgment — guarantee delivery, no lost events
  Exchange types — fanout, direct, topic routing
  Management UI — visual dashboard at :15672
  Industry standard — what real companies actually use

RabbitMQ concepts used in Apex:
  Exchange: apex.events (type: topic)
  Routing keys:
    match.goal       — a goal was scored
    match.kickoff    — match kicked off
    match.fulltime   — match ended
    match.card       — yellow or red card
  Queue: notification.queue (bound to apex.events exchange)

Match service (PUBLISHER):
  services/match-service/src/services/rabbitPublisher.js

  const amqp = require('amqplib')
  
  let channel
  
  async function connect() {
    const conn = await amqp.connect(process.env.RABBITMQ_URL)
    channel = await conn.createChannel()
    await channel.assertExchange('apex.events', 'topic', {
      durable: true
    })
  }

  async function publish(routingKey, data) {
    channel.publish(
      'apex.events',
      routingKey,
      Buffer.from(JSON.stringify(data)),
      { persistent: true }
    )
  }

  module.exports = { connect, publish }

  Usage in livePoller.js:
    publish('match.goal', { matchId, team, player, minute })
    publish('match.kickoff', { matchId, homeTeam, awayTeam })
    publish('match.fulltime', { matchId, score })

Notification service (CONSUMER):
  services/notification-service/src/services/rabbitConsumer.js

  async function startConsuming() {
    const conn = await amqp.connect(process.env.RABBITMQ_URL)
    const channel = await conn.createChannel()
    
    await channel.assertExchange('apex.events', 'topic', {
      durable: true
    })
    
    const q = await channel.assertQueue('notification.queue', {
      durable: true
    })
    
    await channel.bindQueue(q.queue, 'apex.events', 'match.#')
    
    channel.consume(q.queue, async (msg) => {
      const event = JSON.parse(msg.content.toString())
      await handleEvent(msg.fields.routingKey, event)
      channel.ack(msg)       — acknowledge after processing
    })
  }

RabbitMQ Management UI:
  http://localhost:15672 in development
  http://ec2-ip:15672 on AWS (restrict to your IP in security group)
  Login: apex / ${RABBIT_PASSWORD}
  View queues, message rates, consumers — all visual

Key package: amqplib (npm)

--- 5.1 CI WORKFLOW (.github/workflows/ci.yml) ---

name: CI

on:
  push:
    branches: ['*']
  pull_request:
    branches: [main]

jobs:

  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install root deps
        run: npm ci

      - name: Lint all services
        run: npm run lint --workspaces

      - name: Test gateway
        run: npm test
        working-directory: services/gateway

      - name: Test match service
        run: npm test
        working-directory: services/match-service

      - name: Test player service
        run: npm test
        working-directory: services/player-service

      - name: Test ai service
        run: npm test
        working-directory: services/ai-service

      - name: Test user service
        run: npm test
        working-directory: services/user-service

      - name: Test web
        run: npm test
        working-directory: apps/web

  build-check:
    runs-on: ubuntu-latest
    needs: lint-and-test
    steps:
      - uses: actions/checkout@v4

      - name: Build web app
        run: |
          cd apps/web
          npm ci
          npm run build

      - name: Docker build check (gateway)
        run: docker build -t gateway-test ./services/gateway

      - name: Docker build check (web)
        run: docker build -t web-test ./apps/web


--- 5.2 DEPLOY WORKFLOW (.github/workflows/deploy.yml) ---

name: Deploy to AWS

on:
  push:
    branches: [main]

env:
  AWS_REGION: ap-south-1       — Mumbai, closest to Bengaluru
  ECR_REGISTRY: ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.ap-south-1.amazonaws.com

jobs:

  deploy:
    runs-on: ubuntu-latest
    steps:

      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push Gateway image
        run: |
          docker build -t $ECR_REGISTRY/apex-gateway:$GITHUB_SHA \
            ./services/gateway
          docker push $ECR_REGISTRY/apex-gateway:$GITHUB_SHA
          docker tag $ECR_REGISTRY/apex-gateway:$GITHUB_SHA \
            $ECR_REGISTRY/apex-gateway:latest
          docker push $ECR_REGISTRY/apex-gateway:latest

      - name: Build and push Match service image
        run: |
          docker build -t $ECR_REGISTRY/apex-match:$GITHUB_SHA \
            ./services/match-service
          docker push $ECR_REGISTRY/apex-match:latest

      - name: Build and push Player service image
        run: |
          docker build -t $ECR_REGISTRY/apex-player:$GITHUB_SHA \
            ./services/player-service
          docker push $ECR_REGISTRY/apex-player:latest

      - name: Build and push AI service image
        run: |
          docker build -t $ECR_REGISTRY/apex-ai:$GITHUB_SHA \
            ./services/ai-service
          docker push $ECR_REGISTRY/apex-ai:latest

      - name: Build and push User service image
        run: |
          docker build -t $ECR_REGISTRY/apex-user:$GITHUB_SHA \
            ./services/user-service
          docker push $ECR_REGISTRY/apex-user:latest

      - name: Build and push Notification service image
        run: |
          docker build -t $ECR_REGISTRY/apex-notif:$GITHUB_SHA \
            ./services/notification-service
          docker push $ECR_REGISTRY/apex-notif:latest

      - name: Build and push Web app image
        run: |
          docker build -t $ECR_REGISTRY/apex-web:$GITHUB_SHA \
            ./apps/web
          docker push $ECR_REGISTRY/apex-web:latest

      - name: Deploy to EC2 via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ubuntu
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            cd /home/ubuntu/apex
            git pull origin main
            aws ecr get-login-password --region ap-south-1 | \
              docker login --username AWS \
              --password-stdin ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.ap-south-1.amazonaws.com
            docker compose pull
            docker compose up -d --remove-orphans
            docker system prune -f

      - name: Health check after deploy
        run: |
          sleep 30
          curl -f https://apex-sports.com/api/health || exit 1

      - name: Notify success
        if: success()
        run: echo "Deployment successful — $GITHUB_SHA"

SECTION 6: AWS EC2 SETUP STEPS
================================================================

Step 1 — Launch EC2 instance:
  Region: ap-south-1 (Mumbai — closest to Bengaluru)
  AMI: Ubuntu 22.04 LTS (free tier eligible)
  Instance type: t2.micro (free tier, 750hrs/month)
  Storage: 30GB gp2 (free tier max)
  Key pair: create new, download .pem file

Step 2 — Security group rules (inbound):
  Port 22    — SSH          — Your IP only
  Port 80    — HTTP         — 0.0.0.0/0
  Port 443   — HTTPS        — 0.0.0.0/0
  Port 15672 — RabbitMQ UI  — Your IP only
  Port 4000  — Gateway      — 0.0.0.0/0 (or Nginx only)

Step 3 — Assign Elastic IP:
  EC2 → Elastic IPs → Allocate → Associate with instance
  This is your permanent server IP

Step 4 — SSH into EC2 and install Docker:
  ssh -i apex-key.pem ubuntu@YOUR_ELASTIC_IP

  sudo apt update && sudo apt upgrade -y
  sudo apt install -y docker.io docker-compose-plugin git awscli
  sudo usermod -aG docker ubuntu
  newgrp docker

Step 5 — Clone repo on EC2:
  git clone https://github.com/YOUR_USERNAME/apex.git
  cd apex
  cp .env.example .env
  nano .env  — fill in all values

Step 6 — Pull Ollama model on EC2:
  docker compose up ollama -d
  docker exec apex_ollama ollama pull llama3.2
  docker exec apex_ollama ollama create apex-sports \
    -f /path/to/Modelfile

Step 7 — Start all services:
  docker compose up -d
  docker compose ps        — verify all containers running
  docker compose logs -f   — watch logs

Step 8 — SSL certificate with Certbot:
  sudo apt install certbot -y
  sudo certbot certonly --standalone -d apex-sports.com
  Certificates saved to: /etc/letsencrypt/live/apex-sports.com/
  Copy to: ./docker/nginx/ssl/
  Auto-renew: sudo certbot renew --dry-run

Step 9 — Configure Route 53:
  Create hosted zone for apex-sports.com
  Add A record: apex-sports.com → Elastic IP
  Add A record: www.apex-sports.com → Elastic IP
  TTL: 300 seconds

Step 10 — Set up GitHub Actions deployment:
  Add all secrets listed in Section 5.3
  Push to main branch
  Watch Actions tab — first deploy should complete in ~5 min

SECTION 8: DOCKER COMPOSE FOR PRODUCTION (AWS)
================================================================

docker-compose.prod.yml (used on EC2):
Same as dev compose with these differences:

  - No port exposure for individual services
    (only Nginx exposes 80 and 443)
  - Images pulled from ECR instead of built locally
  - Resource limits per container
  - Restart policy: always (not unless-stopped)
  - Logging to CloudWatch

  gateway:
    image: ${ECR_REGISTRY}/apex-gateway:latest
    restart: always
    logging:
      driver: awslogs
      options:
        awslogs-group: /apex/gateway
        awslogs-region: ap-south-1
        awslogs-stream: gateway
    deploy:
      resources:
        limits:
          memory: 150M

  match-service:
    image: ${ECR_REGISTRY}/apex-match:latest
    restart: always
    deploy:
      resources:
        limits:
          memory: 150M

  ollama:
    image: ollama/ollama:latest
    restart: always
    deploy:
      resources:
        limits:
          memory: 600M    — llama3.2 3B needs ~500MB RAM
                            fits in t2.micro with limits set

SECTION 11: ROLLBACK STRATEGY
================================================================

Every deploy tags images with both git SHA and "latest".
ECR keeps last 5 images per repo.

To rollback to previous version on EC2:
  ssh ubuntu@EC2_IP
  cd /home/ubuntu/apex
  
  Pull previous image by SHA (visible in ECR console):
  docker pull ECR_REGISTRY/apex-gateway:PREVIOUS_SHA
  
  Update compose to use previous tag:
  docker compose up -d gateway
  
  Full rollback script (scripts/rollback.sh):
    Takes a commit SHA as argument
    Pulls all service images at that SHA
    Restarts all containers
    Runs health check
