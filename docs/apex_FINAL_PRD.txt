SECTION 1: WHY MICROSERVICES GETS YOU HIRED
================================================================

A monolith says "I can build an app."
Microservices says "I understand how Uber, Nike, and Google
actually ship software."

Each service in Apex:
  - Has one job and does it well
  - Deploys independently — update AI without touching scores
  - Fails independently — if notifications crash, live scores
    still work
  - Scales independently — spin up more match service pods
    during a Champions League final

This is the architecture pattern that senior engineers at
real companies work on every day. Showing it in a portfolio
project puts you in a completely different category from
every other frontend developer applying for the same role.

SECTION 4: API GATEWAY
================================================================

Location: services/gateway/
Port: 4000
Framework: Express.js

This is the ONLY service the clients talk to.
Clients never call individual services directly.

Responsibilities:
  - Verify Firebase ID token on every protected request
  - Route requests to correct downstream service
  - Rate limiting per IP (express-rate-limit)
  - Request logging (morgan)
  - CORS for web and mobile origins
  - WebSocket proxy for live match connections

Routing table:
  /api/matches/*        → match-service:4001
  /api/players/*        → player-service:4002
  /api/ai/*             → ai-service:4003
  /api/users/*          → user-service:4004
  /api/notifications/*  → notification-service:4005
  /ws/live/:matchId     → match-service:4001 (WS)

Files:
  gateway/
    src/
      index.js          — Express app + WebSocket proxy
      middleware/
        auth.js         — Firebase Admin SDK token verify
        rateLimit.js    — express-rate-limit config
        cors.js         — allowed origins
        logger.js       — morgan request logging
      routes/
        proxy.js        — http-proxy-middleware per service
      config/
        services.js     — service URLs from env variables

Key packages:
  express
  http-proxy-middleware — proxy HTTP to downstream services
  firebase-admin        — token verification
  express-rate-limit
  morgan
  cors
  ws                    — WebSocket proxy

SECTION 5: MATCH SERVICE
================================================================

Location: services/match-service/
Port: 4001
Framework: Express.js
Database: MongoDB (matches collection)

This service owns everything about a match.
It is the most critical service in the system.

Endpoints:
  GET  /matches             list live + upcoming matches
  GET  /matches/:id         full match detail
  GET  /matches/:id/stats   detailed stats
  GET  /matches/:id/events  event timeline
  GET  /matches/:id/momentum momentum array
  GET  /live                currently live matches only

WebSocket server (same process):
  ws://match-service:4001/live/:matchId
  Rooms: Map of matchId → Set of ws clients
  Gateway proxies client WS connections here

Live data job (runs inside this service):
  setInterval every 15 seconds
  Fetches from sports APIs (API-Football, TheSportsDB)
  Saves to MongoDB with TTL indexes
  Broadcasts diffs to WebSocket rooms
  Publishes GOAL / CARD events to Redis channel
  "match-events" for notification service to consume

MongoDB schema — matches collection:
  _id           String (external API match ID)
  sport         String
  league        String
  homeTeam      Object { id, name, logo, color }
  awayTeam      Object { id, name, logo, color }
  score         Object { home, away, ht }
  status        String (live / finished / upcoming)
  kickoff       Date
  events        Array of event objects
  stats         Object (possession, shots, passes etc)
  momentum      Array of numbers (per minute)
  fetchedAt     Date (TTL index: expire after 15 seconds)

Files:
  match-service/
    src/
      index.js
      routes/matches.js
      controllers/matchController.js
      models/Match.js
      services/
        sportsApiService.js   — external API wrapper
        livePoller.js         — 15s polling job
        wsHub.js              — WebSocket room manager
        redisPublisher.js     — publish events to Redis

Key packages:
  express
  mongoose
  ws
  axios
  @upstash/redis
  node-cron
  p-queue               — rate limit outbound API calls

SECTION 6: PLAYER SERVICE
================================================================

Location: services/player-service/
Port: 4002
Framework: Express.js
Database: MongoDB (players collection)

Owns all player data and the CSPI engine.

Endpoints:
  GET  /players/:id               player profile
  GET  /players/search?q=         search players
  GET  /players/:id/form          last 10 match form
  GET  /players/:id/career        career arc stats
  GET  /cspi?a=:idA&b=:idB        cross-sport index compare

CSPI Engine (services/cspiEngine.js):
  Pure JavaScript scoring algorithm
  6 dimensions: Speed, Creation, Efficiency,
                Consistency, Impact, Clutch
  Each dimension = formula using sport-specific stats
  All normalized to 0–100 scale
  Final CSPI = weighted average (weights from query params)
  Results cached in MongoDB cspiCache collection for 24h

MongoDB schema — players collection:
  _id           String (external API player ID)
  sport         String
  name          String
  position      String
  nationality   String
  club          Object
  photo         String (Firebase Storage URL)
  careerStats   Array (per season)
  currentForm   Array (last 10 matches)
  cspiScores    Object (pre-calculated dimensions)
  fetchedAt     Date (TTL: 24 hours)

Files:
  player-service/
    src/
      index.js
      routes/players.js
      controllers/playerController.js
      models/Player.js
      services/
        sportsApiService.js

SECTION 7: AI SERVICE — OLLAMA
================================================================

Location: services/ai-service/
Port: 4003
Framework: Express.js
AI Runtime: Ollama (self-hosted, local LLM)

This is the most unique part of the entire stack.
Ollama runs open-source LLMs locally — no API key,
no cost, no rate limits, no data sent to third parties.
This is what companies like Nike and Apple do internally
for data privacy. Showing this in a portfolio is rare.

What is Ollama:
  Ollama is a tool to run LLMs locally on your machine.
  Download from: ollama.ai (free, open source)
  Run a model: ollama pull llama3.2
  It exposes a local HTTP API on port 11434
  The AI service calls this local API

Model selection (all free, run locally):
  Primary:   llama3.2 (3B) — fast, fits 8GB RAM
  Fallback:  phi3 (3.8B)   — Microsoft, very capable
  Sports:    custom Modelfile fine-tuned on sports context

Custom Modelfile (docker/ollama/Modelfile):
  FROM llama3.2
  SYSTEM """
  You are Apex, an expert sports analyst AI.
  You only answer questions about sports and match data.
  You receive structured match data as context.
  Keep answers under 3 paragraphs. Be direct and insightful.
  Never make up statistics. Only use the data provided.
  """
  PARAMETER temperature 0.4
  PARAMETER num_ctx 4096

Endpoints:
  POST /ai/ask         — ask question, SSE streaming response
  POST /ai/summary     — generate match summary
  GET  /ai/health      — check Ollama is running

How /ai/ask works:
  1. Receive: { matchId, question, sessionId }
  2. Call match-service internally to get match data
  3. Load last 5 messages from aiSessions in MongoDB
  4. Build prompt: system + match context + history + question
  5. POST to http://localhost:11434/api/chat with stream:true
  6. Pipe Ollama token stream → SSE to client
  7. Save completed exchange to MongoDB aiSessions

Ollama API call (services/ollamaService.js):
  fetch('http://localhost:11434/api/chat', {
    method: 'POST',
    body: JSON.stringify({
      model: 'apex-sports',
      messages: assembledMessages,
      stream: true
    })
  })
  Read body as ReadableStream, pipe to res (SSE)

SSE to client:
  res.setHeader('Content-Type', 'text/event-stream')
  On each token chunk: res.write('data: ${token}\n\n')
  On done: res.write('data: [DONE]\n\n')

MongoDB schema — aiSessions collection:
  _id          ObjectId
  userId       String (from Firebase UID)
  matchId      String
  messages     Array [{ role, content, timestamp }]
  createdAt    Date
  updatedAt    Date

Files:
  ai-service/
    src/
      index.js
      routes/ai.js
      controllers/aiController.js
      models/AiSession.js
      services/
        ollamaService.js      — Ollama API wrapper + stream
        contextBuilder.js     — assemble match context prompt
        matchClient.js        — internal call to match-service

Key packages:
  express

SECTION 9: NOTIFICATION SERVICE
================================================================

Location: services/notification-service/
Port: 4005
Framework: Express.js
Push: Firebase Cloud Messaging (FCM)
Broker: Redis pub/sub (subscribes to match events)

Owns all push notifications and alert logic.
Does not expose REST endpoints to clients.
Listens to Redis channels published by match-service.

Redis subscriptions:
  Channel: "match-events"
  Message format:
    { type: "GOAL", matchId, team, player, minute }
    { type: "KICKOFF", matchId, homeTeam, awayTeam }
    { type: "FULLTIME", matchId, score }

On receiving a message:
  1. Query user-service: which users follow this team/player?
  2. Get their FCM device tokens from MongoDB
  3. Send FCM push notification via Firebase Admin SDK
  4. Log notification sent

FCM notification payload:
  GOAL: "Messi scored in the 67th minute! Real Madrid 1-0"
  KICKOFF: "Man City vs Arsenal has kicked off"
  FULLTIME: "Final score: PSG 2-1 Bayern Munich"

Weekly digest job (node-cron, runs Sunday 9am):
  Fetch top moments from past week for each user's follows
  Send FCM notification with weekly summary

Files:
  notification-service/
    src/
      index.js
      services/
        redisSubscriber.js  — subscribe to match-events
        fcmService.js       — Firebase Admin SDK push sender
        userClient.js       — internal call to user-service
        digestJob.js        — weekly cron job
      config/
        firebase.js         — Admin SDK init

Key packages:
  express
  firebase-admin
  @upstash/redis

SECTION 12: INTER-SERVICE COMMUNICATION
================================================================

Synchronous (HTTP):
  Gateway → any service: http-proxy-middleware
  AI service → Match service: internal axios call
    http://match-service:4001/matches/:id
  Notification service → User service: internal axios call
    http://user-service:4004/users/by-team/:teamId

Asynchronous (Redis pub/sub via Upstash):
  Match service PUBLISHES to "match-events" channel
  Notification service SUBSCRIBES to "match-events"
  This decouples the two — match service does not know or
  care about notifications at all

Why this matters to hiring managers:
  This is event-driven architecture.
  This is how Uber dispatches rides, how Nike processes
  orders, how Apple pushes App Store updates.
  You built a small version of it.

Service discovery (dev):
  Each service has a known port (4000–4005)
  Gateway reads service URLs from environment variables
  SERVICE_MATCH=http://localhost:4001
  SERVICE_PLAYER=http://localhost:4002
  SERVICE_AI=http://localhost:4003
  SERVICE_USER=http://localhost:4004
  SERVICE_NOTIF=http://localhost:4005

Service discovery (production on Render.com):
  Each service deployed as a separate Render Web Service
  Internal URLs: https://apex-match-svc.onrender.com
  Stored in Gateway environment variables

SECTION 13: LOCAL DEVELOPMENT SETUP
================================================================

docker-compose.yml starts everything:
  Services: gateway, match-svc, player-svc, ai-svc,
            user-svc, notif-svc
  Databases: MongoDB (local), Redis (local)
  Ollama: separate container with GPU passthrough if available

Without Docker (scripts/start-dev.sh):
  Concurrently starts all 5 services + Next.js + Ollama
  Each in its own terminal panel via concurrently package
  npm run dev (root) → starts everything

Ollama setup steps:
  1. Install: curl -fsSL https://ollama.ai/install.sh | sh
  2. Pull model: ollama pull llama3.2
  3. Create custom model: ollama create apex-sports -f Modelfile
  4. Serve: ollama serve (runs on :11434 automatically)
  5. AI service points to http://localhost:11434

Seed database:
  node scripts/seed.js
  Inserts sample matches, players, users into MongoDB
  Allows local dev without hitting sports APIs
