================================================================
APEX SPORTS - DAY 6: PLAYER SERVICE & CSPI ENGINE
================================================================

In Day 6, we focused on building the Player Service, which is responsible for managing athlete data and calculating a unique metric called the Cross-Sport Index (CSPI).

----------------------------------------------------------------
1. PLAYER SERVICE ARCHITECTURE
----------------------------------------------------------------
The Player Service follows the same microservice pattern as the Match Service:
- Express.js for the API framework.
- MongoDB (Mongoose) for the database.
- ES Modules (import/export) for modern JavaScript syntax.
- Middleware: Helmet for security, Morgan for logging, CORS for cross-origin requests.

----------------------------------------------------------------
2. DATA MODELING (PLAYER SCHEMA)
----------------------------------------------------------------
The Player schema is designed to be flexible enough to handle multiple sports while providing deep statistical insights.
- Attributes: Normalized metrics (0-100) like Pace, Shooting, Passing, Dribbling, etc.
- Form: An array of recent match ratings (out of 10) to track current momentum.
- Career Stats: Aggregated data like total matches, goals, and assists.
- CSPI Score: A calculated field representing the player's overall "Power Rating".

----------------------------------------------------------------
3. CSPI (CROSS-SPORT INDEX) ENGINE
----------------------------------------------------------------
This is the core "intellectual property" of the platform. The CSPI Engine uses a weighted algorithm to compare athletes across different sports (e.g., comparing a Footballer's efficiency to a Basketballer's impact).

Algorithm Breakdown:
- Base Attributes (40%): Average of physical and technical attributes.
- Current Form (40%): Weighted average of the last 5 match ratings. High volatility field.
- Career Experience (20%): Logarithmic scale based on total matches played (rewarding longevity).

Interview Tip: "The CSPI Engine solves the 'Comparing Apples to Oranges' problem in sports analytics by normalizing disparate metrics into a unified 0-100 scale."

----------------------------------------------------------------
4. COMPARISON ENDPOINT (GET /api/players/cspi?a=:idA&b=:idB)
----------------------------------------------------------------
We implemented a dedicated endpoint for side-by-side comparisons. 
- It fetches two player profiles.
- Runs them through the CSPI Engine.
- Returns a delta (difference) and identifies the "Leader".
- This is a key feature for the "Compare Players" dashboard in the frontend.

----------------------------------------------------------------
5. SEED DATA & INITIALIZATION
----------------------------------------------------------------
To ensure the system is ready for testing, we added a seeding utility that populates MongoDB with legendary athletes:
- Lionel Messi (Football)
- Cristiano Ronaldo (Football)
- LeBron James (Basketball)
Initial CSPI scores are calculated during the seeding process.

----------------------------------------------------------------
6. GATEWAY INTEGRATION
----------------------------------------------------------------
The API Gateway was already pre-configured to route `/api/players/**` requests to the Player Service on Port 4002. This ensures the frontend only interacts with a single entry point.

----------------------------------------------------------------
INTERVIEW PREP - KEY QUESTIONS:
----------------------------------------------------------------
Q: Why use a logarithmic scale for Career Experience?
A: Because the difference between 0 and 100 matches is massive (gaining experience), but the difference between 1000 and 1100 matches is marginal in terms of skill development. Log scales dampen the impact of extremely high numbers.

Q: How do you handle schema differences between Sports?
A: We use a generic 'attributes' object and 'careerStats' object. While the fields (goals vs points) might vary in meaning, the structure remains consistent for the CSPI engine to process.

Q: Why separate Match Service and Player Service?
A: Scaling. Match data is high-frequency and volatile (live scores), while Player data is relatively stable but requires complex calculations (CSPI). Separating them allows us to scale the Match service independently during peak live events.
